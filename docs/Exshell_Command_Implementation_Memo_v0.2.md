# Exshell コマンド実装方針メモ v0.2

## 1. 文書目的

本メモは、Exshell 外部仕様書 v0.2 と内部設計案 v0.2 に基づき、実装着手時の具体的な進め方、優先順位、注意点を整理するためのものである。

v0.2 では主利用環境を WSL ターミナルへ変更したため、v0.1 時点よりも以下を強く意識する。
- WSL から Windows 実行ファイルを呼ぶ運用
- パイプ、リダイレクト、process substitution との相性
- WSL パスと Windows パスの変換
- stdout / stderr の純度維持

---

## 2. 実装の基本戦略

### 2.1 実装思想
Exshell は多機能シェルではない。
Exshell は Excel と標準入出力をつなぐ専用ブリッジである。

したがって、実装方針は次のようにする。
- 汎用テキスト処理は実装しない
- Excel 操作に必要な最小機能を堅く作る
- stdout に無駄な文言を流さない
- WSL と組み合わせた時に自然に使えることを最優先にする

### 2.2 最初に守る原則
- `ecat` は本文だけを stdout に出す
- `cate` は stdin を素直に受ける
- エラーは stderr に出す
- 成功/失敗は終了コードで判別可能にする
- Save / Close / Quit は呼ばない

---

## 3. 実装優先順位

## 3.1 第1段階: 共通基盤
最初に以下を作る。
- コマンドライン解析共通処理
- セッション JSON 読書き
- パス正規化
- Excel 接続共通処理
- シート/Shape 識別子解析
- 終了コード体系

これを先に作る理由は、以後の各コマンドで共通利用されるためである。

## 3.2 第2段階: 読み取り系
以下の順に実装する。
1. `eopen`
2. `einfo`
3. `els`
4. `ecat`

特に `ecat` は WSL 連携の中心になるため、早めに完成させる。

## 3.3 第3段階: 書き込み系
以下の順に実装する。
1. `cate` 上書き
2. `cate --append`

最初は上書きのみでもよいが、仕様確定済みのため追記も早めに入れる。

## 3.4 第4段階: 差分補助
- `ediff`

ただし `ediff` は最後でもよい。
理由は、`ecat` が完成していれば手動で `diff <(ecat ...) <(ecat ...)` の検証が先に行えるためである。

---

## 4. コマンド別実装方針

## 4.1 eopen 実装方針

### 4.1.1 先に作るべき理由
- セッションの起点になる
- Excel 接続ロジックのベースになる

### 4.1.2 実装ポイント
- 引数で受けたパスを Windows 絶対パスへ変換
- ファイル存在確認
- Excel.Application を取得または起動
- 同一ブックが既に開かれているか判定
- 必要に応じて Open
- セッション保存

### 4.1.3 注意点
- WSL パス `/mnt/d/...` を変換できること
- 既存 Excel インスタンス取得が失敗した場合のフォールバックを明確にすること
- ブックの保存状態は変更しないこと

---

## 4.2 ecat 実装方針

### 4.2.1 重要度
最重要コマンドの一つ。
これが自然に動けば、WSL での利用価値が一気に上がる。

### 4.2.2 実装ポイント
- セッションから WorkbookPath / DefaultSheet を取得
- `Sheet:ShapeName` を解決
- Shape の文字列を取得
- stdout にそのまま出力

### 4.2.3 出力ルール
- 不要な見出しを付けない
- 「読み取り成功」等の文言を stdout に出さない
- 必要なら stderr にだけ補助情報を出す

### 4.2.4 先行確認コマンド例
```bash
ecat Main:src | nl
ecat Main:src | grep "TODO"
cat <(ecat Main:left)
```

これらが自然に成立することを確認する。

---

## 4.3 cate 実装方針

### 4.3.1 実装ポイント
- stdin 全文読取
- 書込対象 Shape 解決
- 上書き/追記を選択
- Shape テキストを更新

### 4.3.2 追記モード
追記は次の扱いを最初に決める。
- そのまま連結するか
- 必要に応じて改行を挟むか

初期版では、まずは「素直な連結」を基本とし、必要に応じて呼び出し側で改行を含める方が扱いやすい。

### 4.3.3 運用確認例
```bash
printf 'line1\nline2\n' | cate Main:in
printf 'append text\n' | cate --append Main:in
```

### 4.3.4 注意点
- stdin が空でも仕様上の扱いを決める
- 書込後に Save しないことを崩さない
- Excel 画面上で変化が見えることを手動確認する

---

## 4.4 els 実装方針

### 4.4.1 実装ポイント
- 対象 Worksheet の Shapes を列挙
- テキスト取得対象のみ抽出
- 名前一覧を stdout 出力

### 4.4.2 価値
- Shape 名運用を支える基礎コマンド
- `ecat` / `cate` 前の確認手段になる

### 4.4.3 確認例
```bash
els
els --sheet Main
```

---

## 4.5 ediff 実装方針

### 4.5.1 位置づけ
`ediff` は diff ロジックを自前実装するためのコマンドではない。
内部で一時ファイルを作成し、WSL の `diff` を呼ぶ簡便コマンドとして実装する。

### 4.5.2 実装ポイント
- 左右テキスト取得
- 一時ファイル生成
- UTF-8 / LF で保存
- `wsl diff temp1 temp2` 実行
- stdout / stderr / exit code を中継
- temp 削除

### 4.5.3 補足
WSL 主体運用では、上級利用者は次のように直接 `diff` を使える。
```bash
diff <(ecat Main:left) <(ecat Main:right)
```
そのため `ediff` はショートカットであることを意識し、作り込みすぎない。

## 5. パス変換実装の進め方

## 5.1 最初に作る変換対象
- `D:\work\sample.xlsx`
- `/mnt/d/work/sample.xlsx`

まずこの2系統に限定して実装し、安定させる。

## 5.2 変換規則
WSL パスの基本変換:
- `/mnt/c/...` → `C:\...`
- `/mnt/d/...` → `D:\...`

## 5.3 注意点
- スペースを含むパス
- 日本語パス
- 相対パス

相対パスは初期版では無理に広く対応せず、絶対パス優先で進める方が安全。

---

## 6. COM 実装の注意点

## 6.1 最重要事項
Excel COM は動いて見えても後で不安定になりやすい。
したがって次を徹底する。
- COM オブジェクトの寿命を短く保つ
- finally で解放する
- 深いネストで参照を持ち回りしすぎない

## 6.2 Shapes 走査時の注意
`foreach` で回すと見えない COM 参照が増えることがあるため、必要に応じて index ベースアクセスも検討する。

## 6.3 TextFrame 系の注意
Shape によっては文字列取得不能なものがある。
そのため、判定ロジックを共通メソッドに切り出しておく。

---

## 7. stdout / stderr 実装方針

## 7.1 stdout の原則
WSL 側のパイプ処理のため、stdout は純粋であるべき。

悪い例:
```text
Reading textbox...
Hello
```

良い例:
```text
Hello
```

## 7.2 stderr の原則
エラー時は stderr に必要十分な説明を書く。

例:
```text
Shape not found: Main:src
```

## 7.3 実装上の注意
- ログやデバッグ文を stdout に混ぜない
- 将来 verbose モードを入れる場合も stderr 側へ出す

---

## 8. 文字コード・改行の実装方針

## 8.1 一時ファイル
- UTF-8
- LF

## 8.2 ecat
- 文字化けしないことを最優先
- 日本語テキストで WSL コマンドと接続確認する

## 8.3 cate
- stdin の改行をどう Excel に反映するか確認する
- Excel 表示で崩れがないか手動確認する

## 9. 実装単位の切り方

単一の巨大 EXE にサブコマンドを持たせるか、`eopen.exe`, `ecat.exe` のように分割するかは早めに決める必要がある。

### 9.1 推奨
初期は単一 EXE + サブコマンドでもよいが、WSL からの使いやすさを重視するなら、最終的にはコマンド名ごとに起動できる形が望ましい。

例:
- `exshell.exe ecat Main:src`
- `ecat.exe Main:src`

利用体験としては後者が強い。

### 9.2 現実的な案
内部実装は共通 DLL に寄せ、薄いエントリポイント EXE を複数用意する。

## 10. MVP 完了条件

以下が通れば、v0.2 の MVP として十分価値がある。

### 10.1 必須
- `eopen` で Windows パス / WSL パスの両方を受けられる
- `els` で Shape 一覧が取れる
- `ecat` が WSL パイプで自然に使える
- `cate` が stdin から書き込める
- `cate --append` が使える
- `ediff` が WSL diff を呼べる
- Save / Close / Quit をしない

### 10.2 WSL 利用確認
次が実際に動くこと。
```bash
ecat Main:src | nl
ecat Main:src | grep "TODO"
printf 'abc\n' | cate Main:dst
diff <(ecat Main:left) <(ecat Main:right)
```

## 11. 実装時の落とし穴

## 11.1 相対パスに深入りする
最初から完璧にやろうとすると複雑化しやすい。
まずは絶対パス優先で進める。

## 11.2 stdout 汚染
少しの説明文でもパイプ運用が壊れる。
特に `ecat` は厳格に保つ。

## 11.3 COM 解放漏れ
Excel プロセス残留の原因になる。
共通解放パターンを最初に作る。

## 11.4 ediff を作り込みすぎる
本命は WSL の `diff` であり、Exshell は橋であることを忘れない。

## 11.5 PowerShell 運用前提を引きずる
v0.2 では主戦場は WSL である。
利用例、テスト例、出力思想もそれに合わせる。

## 13. まとめ

v0.2 の実装では、Exshell を Excel と WSL の橋渡しに集中させることが最重要である。
特に `ecat` / `cate` の品質が全体の価値を決める。

実装時は次を常に優先する。
- WSL で自然に使えること
- stdout を汚さないこと
- COM を安全に扱うこと
- path / encoding / newline を明示的に扱うこと

`ediff` は便利だが主役ではない。
主役は、`ecat | grep`, `ecat | nl`, `ecat | diff`, `... | cate` という Unix 的な接続性である。
